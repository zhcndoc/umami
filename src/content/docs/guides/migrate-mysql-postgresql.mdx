---
title: 从 MySQL 迁移到 PostgreSQL
---

作为 Umami v3 的一部分，我们将停止支持 MySQL 作为数据库选项，并统一使用 PostgreSQL。
此更改确保所有环境中数据层更加一致且具备更好的可扩展性。

## 数据迁移

本文档概述了从 MySQL 迁移到 PostgreSQL 的过程，包括：

- 准备 MySQL 转储文件或 CSV 文件。
- 验证数据完整性和应用兼容性。

目标是实现无缝过渡，同时保留所有数据并确保最小的停机时间。你可以探索一些开源工具，如
`pgloader` 和 `pg_chameleon`。下面介绍的过程是将数据纯粹迁移到新的 v2 PostgreSQL 数据库，
使此方法尽可能简单且无偏向性。

## 要求

- MySQL 数据库模式必须与最新的 v2 版本（v2.19.0）同步。
- 两个数据库的凭据：

```shell
MySQL: mysql://user:password@host:port/dbname

PostgreSQL: postgresql://user:password@host:port/dbname
```

## 使用 PostgreSQL 数据库安装 Umami

1. 在 PostgreSQL 服务器上安装 Umami (v2.19.0) 。安装过程文档见 [这里](http://localhost:5001/docs/install)。
2. 清空以下表。这些表会通过 MySQL 导入进行填充。

```sql
truncate table "_prisma_migrations";
truncate table "user";
```

## 执行迁移

### MySQL 转储文件

对于较小的数据库（少于 100 万条记录），你可以使用 MySQL 转储文件。官方文档见 [这里](https://dev.mysql.com/doc/refman/8.4/en/mysqldump.html)。

1. 生成 MySQL 转储文件。

```shell
mysqldump --no-create-info --default-character-set=utf8mb4 --quick --skip-add-locks \
-u username -p mydb > mydbdump.sql
```

2. 将反引号替换为双引号，使之兼容 PostgreSQL。

```shell
sed -i 's/`/"/g' mydbdump.sql
```

3. 使用 psql 执行转储文件

```shell
psql -U username -d mydb < mydbdump.sql
```

Docker 版

```shell
docker exec -i my-container psql -U username -d mydb < mydbdump.sql
```

现在你可以升级到 Umami (v3.0)。

### 生成 CSV 文件

对于大型数据库，转储文件可能过大且插入语句效率低下。

1. 为所有必要并含有数据的表生成 MySQL CSV 文件。

在 MySQL shell 中执行：

```sql
SELECT *
INTO OUTFILE '/tmp/website_event.csv'
FIELDS TERMINATED BY ','
ENCLOSED BY '"'
LINES TERMINATED BY '\n'
FROM website_event;
```

**注意**

- 文件路径（/tmp/users.csv）是在 MySQL 服务器的文件系统中，不一定是你的宿主机。
- 确保 MySQL 有权限写入该路径（secure_file_priv 可能会限制写入权限）。

2. 使用 `scp` 或 `cp` 将文件从 MySQL 服务器移动到 psql 可访问的位置。

3. 使用 psql 的 `\COPY` 命令将 CSV 导入 PostgreSQL 数据库。每张表重复执行该命令。

```shell
psql -U postgres -d mydb -c "\COPY website_event FROM 'website_event.csv' CSV NULL '\N'"
```

**注意**

- 如未使用默认模式，请包含模式名称。
