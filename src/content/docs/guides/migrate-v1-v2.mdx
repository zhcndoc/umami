---
title: 从 v1 迁移到 v2
---

Umami v2 引入了重新设计的架构以及一些重大变更。

## 重大变更

- API 端点发生了变化，并新增了一些端点。详见 [API](/docs/api)。
- 追踪脚本名称由 `umami.js` 更改为 `script.js`。
- collect API 端点由 `/api/collect` 重命名为 `/api/send`。
- 追踪器不再使用 CSS 类名触发事件，而是改用 HTML data 属性。详见 [跟踪事件](/docs/v2/track-events)。
- 全局 `umami` JavaScript 对象的方法已更改，现统一为单个 `.track()` 方法，并有新的事件数据发送方式。详见 [追踪器函数](/docs/v2/tracker-functions)。
- `TRACKER_SCRIPT_NAME` 环境变量不再自动添加 `.js` 后缀。详见 [环境变量](/docs/v2/environment-variables)。

## 数据迁移

由于架构变更，你的 v1 数据库中的数据需要转换成 v2 格式。
为帮助迁移，我们创建了 `@umami/migrate-v1-v2` 脚本，可自动迁移所有数据。

### 要求

- 数据库架构必须与最新的 v1 版本（v1.4.0）保持同步。脚本会查询 prisma 的 migrations 表以确保已运行最新迁移。

### 重要事项

- 使用前请备份目标数据库。若迁移过程中断，可能导致数据丢失。
- 对于大数据集用户（500 万条及以上），迁移可能耗时较长，请合理安排时间。
- 脚本不会迁移任何事件数据至 v2。
- 迁移完成后，脚本会询问是否删除你的 v1 表。
- 若发现 `event_data` 表有数据，将重命名为 `v1_event_data`，但不会删除。

### 故障排查

- 如果你的 `DATABASE_URL` 设置为 localhost 且迁移脚本无法连接数据库，尝试改为 IP 地址，比如：`127.0.0.1`。

## 运行迁移

有两种方式运行迁移脚本。

### 在 Umami 文件夹内运行

适用于有应用文件夹终端访问权限的情况。
确保应用已构建完成，否则先执行 `yarn build`。

```shell
cd umami
npx @umami/migrate-v1-v2@latest
```

### 独立运行

适用于无法访问应用文件夹的情况，比如部署在 Vercel 或 Netlify。

**安装**

```shell
git clone https://github.com/umami-software/migrate-v1-v2.git
cd migrate-v1-v2
yarn install
yarn build
```

**配置**

创建 `.env` 文件，定义以下变量：

```
DATABASE_URL={连接 URL}
```

**运行**

```shell
yarn start
```

## Docker 迁移

进入你正在运行的 Docker 容器。可以通过 `docker ps` 获取容器名称。

```shell
docker exec -ti -u 0 <应用容器名称> sh
```

运行迁移脚本。

```shell
npx @umami/migrate-v1-v2@latest
```
