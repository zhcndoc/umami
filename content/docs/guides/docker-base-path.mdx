---
title: Setting a Base Path for Docker
---

By default, Umami is served from the root path (`/`). If you need to host Umami under a subpath (e.g. `/umami`),
you can use the pre-built base path image variant. This is useful when running Umami behind a reverse proxy alongside
other applications on the same domain.

## How it works

The `BASE_PATH` build argument is declared in the Dockerfile and passed into the Next.js build. Because Next.js
bakes the base path into the compiled output, the value **cannot be changed at runtime** — it must be set when the
image is built.

```dockerfile
ARG BASE_PATH=""
ENV BASE_PATH=$BASE_PATH
```

The resulting image is labeled so you can inspect which base path it was built with:

```shell
docker inspect <image> --format '{{ index .Config.Labels "org.opencontainers.image.base-path" }}'
```

## Pre-built image variants

Every release automatically builds two image variants via a build matrix in the CI pipeline:

| Variant | Base path | Tag suffix | Example tag |
| --- | --- | --- | --- |
| Standard | _(none)_ | _(none)_ | `umamisoftware/umami:3.0.0` |
| Base path | `/umami` | `-base` | `umamisoftware/umami:3.0.0-base` |

Both variants are pushed to [Docker Hub](https://hub.docker.com/r/umamisoftware/umami) and
[GHCR](https://github.com/umami-software/umami/pkgs/container/umami) on every release.

To use the base path variant, pull the image with the `-base` suffix:

```shell
docker pull umamisoftware/umami:latest-base
```

## Running the base path image

Run the container as usual. Umami will be available at `http://localhost:3000/umami`.

```shell
docker run -e DATABASE_URL=postgresql://user:password@host:5432/umami -p 3000:3000 umamisoftware/umami:latest-base
```

## Building a custom image

If you need a different subpath, pass `BASE_PATH` as a build argument when running `docker build`:

```shell
docker build --build-arg BASE_PATH=/analytics -t umami:custom .
```

## Triggering a release build

The workflow runs automatically on every version tag push and builds both variants. You can also trigger it
manually from **Actions → Create docker images → Run workflow** with these inputs:

| Input | Description |
| --- | --- |
| `version` | Optional version override (e.g. `3.0.0`). Defaults to the triggering tag. |
| `include_latest` | Whether to also push a `latest` / `latest-base` tag. |

## Reverse proxy configuration

When hosting under a subpath you will need to configure your reverse proxy to forward requests correctly.

**Nginx example**

```nginx
location /umami/ {
    proxy_pass http://localhost:3000/umami/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}
```

**Notes**

- The trailing slash on both `location` and `proxy_pass` is important.
- The tracking script URL will also include the base path (e.g. `/umami/script.js`).
- Update any existing website configurations in Umami to use the full script URL with the subpath.
